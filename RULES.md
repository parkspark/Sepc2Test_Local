# TC 작성 규칙 (RULES)

TC를 작성하는 모든 반복이 따라야 하는 유일한 규칙 문서. 루프 진행 절차는 `PROMPT.md`에, 작성 규칙은 여기에 있다.

## 1. 역할과 원칙
- 너는 숙련된 QA 엔지니어다. 게임 기획문서(Spec)를 분석해 프로젝트의 기존 TC 형식에 맞는 신규 TC를 작성한다.
- **블랙박스 테스트 관점만** 취한다. 클라이언트/서버 코드, 내부 구현은 참조·판단하지 않는다.
- **스펙(기획문서)이 유일한 기준이다.** 기존 TC와 기획문서 내용이 다르면 무조건 기획문서를 따른다. 기존 TC는 형식·스타일 참고용일 뿐, 내용의 근거로 삼지 않는다.
- **기획문서에 없는 내용을 추측해서 TC를 만들지 않는다.** 근거가 부족하면 §6 기준에 따라 의문점 목록으로 분류한다.
- 생성된 TC는 사람이 검수 후 반영하며, CSV는 DB 저장을 전제로 한다.
- 섹션별 테스트 자동화에 활용할 예정이므로 **각 섹션은 독립적으로 완결된 TC 집합**을 가져야 한다.

## 2. 입력
- `input/`의 기획문서 PDF: 분석 대상.
- `input/`의 기존 TC(xlsx): 작성 스타일(용어, 문장 톤, 분류 체계) 참고용. 구체 내용(수치·기능)은 작업 대상이 아니므로 스타일 참고 이상으로 사용하지 않는다.
- 스타일 요약은 Phase 0이 `state/style_guide.md`로 추출해 둔다. 각 반복은 원본 xlsx 대신 이것을 읽는다.

## 3. TC 범위 규칙
- 정상 케이스 외에 **예외/경계값 케이스**도 포함한다.
- **Appendix 등 표(테이블) 수치**: 표의 각 행(레벨/단계 등)마다 별도 TC를 1개씩 생성한다. (예: 레벨별 목표 점수 테이블 → 각 레벨의 목표 점수 도달 여부를 확인하는 TC를 레벨 수만큼 생성)
- **아트/리소스 스펙도 꼼꼼하게 TC화한다.** 색상, 이펙트 종류/방향, 연출(애니메이션 흐름), 이미지 상태 변화, 사이즈 등 기획서에 명시된 시각 스펙을 확인 가능한 형태로 만든다.
  - 예: "이펙트 종류는 총 3가지(대각선2, 수평베기1)" → "더미 터치 시 타격 이펙트가 3종(대각선 2종, 수평베기 1종) 중 랜덤으로 출력된다."
- **"OO와 동일" 재사용 표현**: 기획서에 "검술 훈련과 동일" 같은 표현이 있어도 해당 섹션에 TC를 **온전히 다시 작성**한다. 원본 섹션을 참조만 하고 생략하는 것을 금지한다. (섹션별 자동화를 위한 자기완결성)

## 4. TC CSV 형식
- 컬럼(9개 고정): `No, 대분류, 중분류, 소분류, 테스트 항목, 사전조건, 테스트 스텝, 기대결과, 비고`
  - `Test Level` 컬럼은 사용하지 않는다.
- `No`: 협업 시 지칭용 인덱스. 1부터 시작하는 일련번호. (섹션 파일에서는 섹션 내 1부터, 최종 병합 시 전체 재부여)
- 인코딩: **UTF-8 BOM 포함** (엑셀 한글 깨짐 방지)
- **fill-down**: 모든 행에 값을 채운다. 대분류/중분류/소분류/테스트 항목이 같은 그룹이라도 각 행에 반복 기재한다 (DB 저장·프로그램 처리용). 엑셀식 셀 병합 표현 금지.
- 사전조건: 실제 조건이 있을 때만 작성. 없으면 **빈 값** (`-` 같은 placeholder 금지).
- 분류는 기획문서의 목차·섹션 구조를 따른다: 대분류 = 기능/미니게임 단위, 중분류 = 화면/PU 단위, 소분류 = 화면 내 기능 그룹 단위.

### 문장 스타일 (`state/style_guide.md`와 함께 적용)
- 테스트 스텝: "~을 확인한다.", "~을 터치한다." 등 행위 서술형, 마침표로 종결. 여러 단계면 "1. ~ 2. ~" 번호 매김.
- 기대결과: "~된다.", "~출력된다." 등 결과 서술형, 마침표로 종결.
- 여러 UI 요소를 한 번에 검증할 때: 기대결과 안에 "다음과 같은 요소가 출력된다. 1. ~ 2. ~" 형태로 번호 나열.
- 용어는 기존 TC의 약어를 그대로 사용: `btn`(버튼), `PU`(팝업), `Lv.`(레벨) 등. **새 약어를 임의로 만들지 않는다.**

### 비고 컬럼
- 기본은 빈 값. 해당 TC 자체에 특이점이 있는 경우에만 작성한다. 허용 용도:
  - 기획서에 "추후 변경 가능/조정 가능"으로 표기된 수치 기반 TC → `기획서에 변경 가능성 명시 — 확정값 확인 필요`
  - 그 외 검수자가 반드시 알아야 할 TC 단위 특이사항
- 일반 TC에는 아무것도 적지 않는다. 기획서 출처 표기, 설명 반복 등으로 채우지 않는다.

## 5. 의문점 목록 형식
- 섹션 작업 시 `work/questions/<번호>_<섹션명>.md`에 기록. 최종 병합 파일명: `의문점_[기획서명].md`
- 기획서에 등장하는 순서대로 정리한다.
- 각 항목에 출처(기획서 섹션/슬라이드 번호)를 명시한다.
- §6 기준에 해당하는 경우에만 포함한다.
- TC로 이미 생성되고 비고에 특이점이 기록된 항목("추후 변경 가능" 수치 등)은 중복 기재하지 않는다. 의문점 목록은 **TC로 만들 수 없었던 항목 전용**이다.

예시:

```markdown
## 100 궁술 훈련 - 훈련장 로비 - UI - 1. 군량미
- 소유한 군량미 개수의 최대값(cap)이 있는가? (출처: 슬라이드 5)
- 표시 자릿수의 최대치는? 최대치 초과 시 "99999+" 형태로 표기되는가? (출처: 슬라이드 5)
```

## 6. 애매모호함 판단 기준
다음 중 하나 이상이면 TC로 만들지 않고 의문점 목록으로 분류한다:
- 수치/범위/최대값 등 구체적 값이 기획서에 명시되지 않은 경우 (예: 재화 보유 개수 상한, 자릿수 표기 규칙)
- 동일 항목에 대해 기획서 내 설명이 서로 모순되는 경우
- 예외 상황(에러, 실패, 취소, 네트워크 끊김 등)의 처리 방식이 기술되지 않은 경우
- 여러 화면/기능 간의 우선순위나 상호작용이 정의되지 않은 경우

단, 다음은 의문점이 아니라 **TC로 처리**한다:
- "추후 변경될 수 있음", "추후 조정 가능" 등으로 표기된 수치: 현재 명시된 값 기준으로 TC를 생성하고, 그 TC의 비고에 `기획서에 변경 가능성 명시 — 확정값 확인 필요`라고 기록한다. (값 자체는 존재하므로 TC 생성이 가능하고, 비고 표기로 추후 수치 변경 시 영향받는 TC를 필터링할 수 있다)

## 7. 최종 검증 체크리스트 (Phase 2에서 항목별 확인)
- [ ] 기획문서의 모든 섹션·항목이 TC 또는 의문점 중 하나로 커버됨 (누락 없음)
- [ ] 아트/리소스 스펙이 기획서에 명시된 범위에서 빠짐없이 TC화됨
- [ ] "OO와 동일" 항목이 생략 없이 각 섹션에 자기완결적으로 작성됨
- [ ] 분류 체계·문장 스타일이 기존 TC(style_guide)와 일치함
- [ ] 비고는 정의된 특이점 용도로만 사용되고 일반 TC에서는 비어 있음
- [ ] 각 의문점에 기획서 출처가 있음
- [ ] CSV가 fill-down 형태로 DB에 바로 적재 가능 (`validate_csv.py --final` 통과)
