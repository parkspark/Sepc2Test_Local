# TC 자동 생성 — 랄프 루프 반복 지시서

> 이 문서는 Claude Code 무인 루프 경로의 지시서다. 로컬 LLM(Ollama) 경로는
> `scripts/local_pipeline.py`가 이 문서의 Phase 0/1/2 정의·형식을 그대로 구현해
> 파이썬이 직접 수행한다 — 절차의 근거는 여전히 이 문서다.

너는 반복 실행되는 루프의 **한 번의 반복(iteration)**이다. 이전 반복의 결과는 오직 파일로만 전달된다.
**이번 반복에서는 딱 한 단위의 작업만 수행하고 종료한다.** 욕심내서 여러 작업을 한 번에 처리하지 않는다.

## 매 반복 시작 절차
1. `RULES.md`를 읽는다. — TC 작성에 관한 모든 판단 기준.
2. `state/PROGRESS.md`를 읽는다.
   - 파일이 없으면 → 이번 반복은 **Phase 0(초기화)**.
   - 있으면 → 위에서부터 첫 번째 미완료 `[ ]` 항목 **하나**가 이번 반복의 작업.
3. `state/NOTES.md`가 있으면 읽는다. — 이전 반복들이 남긴 팁·주의사항.
4. **끊김 복구**: PROGRESS에 체크되지 않았는데 산출물이 이미 존재하는 섹션이 있으면 직전 반복이 중간에 끊긴 것이다. 해당 산출물(`work/tc/`, `work/questions/`의 그 섹션 파일)을 삭제하고 그 섹션부터 새로 작업한다. 부분 산출물은 신뢰하지 않는다.

## Phase 0 — 초기화 (최초 1회)
1. `input/`을 확인한다: 기획문서 PDF 1개(분석 대상), 기존 TC xlsx(스타일 참고)가 있어야 한다. 없거나 무엇이 기획서인지 판단 불가하면 `state/NEEDS_HUMAN`에 사유를 쓰고 종료.
2. 기획문서의 목차·섹션 구조를 파악해 `state/sections.md`를 작성한다:
   - 첫 줄에 `기획서명: <PDF 파일명(확장자 제외)>`
   - 섹션 목록: `번호(001, 002…) | 섹션 제목 | 페이지(슬라이드) 범위 | 분류 후보(대분류/중분류)`
   - 섹션은 **중분류(화면/PU) 수준**으로 쪼갠다 — 한 반복에서 무리 없이 끝낼 크기. 너무 크면 더 잘게 나눈다. Appendix의 표는 관련 섹션에 포함시키고 그 사실을 함께 적는다.
   - PDF를 직접 읽기 어려우면 python(pdfplumber, pdftoppm 등)으로 `work/spec/`에 페이지별 텍스트·이미지를 추출해 두고, 이후 반복이 그것을 읽게 한다. 이 사실을 `state/NOTES.md`에 기록한다.
3. 기존 TC xlsx를 읽고(openpyxl 등) `state/style_guide.md`를 작성한다:
   - 실제 파일에서 발견된 약어 목록 (`btn`, `PU`, `Lv.` 등 — 발견된 것만)
   - 테스트 스텝/기대결과 문장 패턴 (대표 예시 5~10개 인용)
   - 대/중/소분류 구성 방식
   - ※ 내용(수치·기능)은 옮기지 않는다 — 스타일만. (스펙이 유일한 기준)
4. 아래 템플릿으로 `state/PROGRESS.md`를 생성한다 (섹션 목록은 sections.md 순서대로):

   ```markdown
   # PROGRESS
   ## Phase 0 — 초기화
   - [x] sections.md / style_guide.md 작성
   ## Phase 1 — 섹션별 TC 생성
   - [ ] 001 <섹션명> (p.X–Y)
   - [ ] 002 <섹션명> (p.X–Y)
   ## Phase 2 — 병합·최종 검증
   - [ ] CSV 병합 → output/TC_<기획서명>.csv (No 전체 재부여)
   - [ ] 의문점 병합 → output/의문점_<기획서명>.md
   - [ ] 최종 검증 (validate_csv.py --final + RULES §7 체크리스트 → state/coverage_report.md)
   - [ ] state/DONE 생성
   ```

5. git 저장소면 커밋(`init: 섹션 인벤토리/스타일 가이드`) 후 종료.

## Phase 1 — 섹션 하나의 TC 생성
1. PROGRESS에서 선택한 섹션의 기획서 페이지**만** 읽는다 (전체를 다시 읽지 않는다). `state/style_guide.md`도 읽는다.
2. 섹션의 기능/UI/아트 요소를 먼저 목록화한 뒤, `RULES.md`에 따라 TC와 의문점을 작성한다.
3. 산출물 2개를 만든다:
   - `work/tc/<번호>_<섹션명>.csv` — 헤더 포함, UTF-8 BOM, No는 섹션 내 1부터 (병합 시 전체 재부여됨).
   - `work/questions/<번호>_<섹션명>.md` — 의문점이 없어도 `(없음)`이라고 쓴 파일을 만든다 (검토 완료의 증거).
4. 검증: `python scripts/validate_csv.py work/tc/<파일>` 실행. ERROR가 0이 될 때까지 수정 후 재실행. WARN은 타당한지 검토하고 판단을 NOTES.md에 남긴다.
5. 자가 점검: 이 섹션의 모든 요소가 TC 또는 의문점 중 한쪽에 들어갔는가? "OO와 동일" 항목을 자기완결적으로 풀어썼는가? (누락·참조 생략 금지)
6. PROGRESS에서 해당 섹션을 `[x]`로 갱신하고, git 저장소면 커밋(`tc: <섹션명>`) 후 종료.
- 같은 섹션이 2회 연속 실패하면(NOTES.md 기록으로 확인) PROGRESS에 `[!] BLOCKED — 사유`로 표기하고 다음 반복으로 넘긴다.

## Phase 2 — 병합·최종 검증 (Phase 1이 전부 [x]일 때)
PROGRESS의 Phase 2 항목도 한 반복에 하나씩 처리한다.
- **BLOCKED 섹션이 남아 있으면** `state/NEEDS_HUMAN`에 목록과 사유를 쓰고 종료 (러너가 멈춘다).
- **CSV 병합**: `work/tc/*.csv`를 섹션 번호 순으로 합쳐 `output/TC_<기획서명>.csv` 생성. 헤더는 1회만, No는 1부터 전체 재부여, UTF-8 BOM. python 스크립트로 처리한다 (수작업 금지).
- **의문점 병합**: `work/questions/*.md`를 순서대로 합쳐 `output/의문점_<기획서명>.md` 생성. `(없음)` 파일은 제외, 중복 의문점은 출처를 병기해 하나로 합친다.
- **최종 검증**: `python scripts/validate_csv.py --final output/TC_*.csv` 통과 필수. 이어서 RULES.md §7 체크리스트를 항목별로 확인하고 결과를 `state/coverage_report.md`에 기록한다. 불통과 항목이 있으면 PROGRESS에 수정 작업 항목을 추가하고 종료 (다음 반복이 처리).
- 모두 통과하면 `state/DONE`을 생성한다 (내용: 산출물 경로, 총 TC 수, 의문점 수, 검수자가 확인할 사항 요약). 러너가 이 파일을 보고 정지한다.

## 공통 규칙
- **한 반복 = 한 작업.** 완료된 작업은 다시 하지 않는다.
- 진행 추적은 `state/PROGRESS.md`로만 한다. 다른 곳에 TODO를 만들지 않는다.
- 작업 중 얻은 교훈(기획서 구조 특이점, 파싱 방법, 스타일 발견, 실패 기록)은 `state/NOTES.md`에 짧게 추가해 다음 반복에 전달한다.
- 사람 개입 없이는 진행 불가한 상황 → `state/NEEDS_HUMAN`에 사유를 쓰고 종료.
- 종료 직전 마지막 출력: `[반복 요약] 완료: <이번 작업> | 다음: <다음 항목> | 남은 섹션: <N>`
